# 开发队新干培养方案

校学生处技术开发团队新干培训，通过三个月的"老带新"项目式学习（Project-Based Learning, PBL）夯实基础、提升专业技能。

## 培养方案

### 总体目标

- **短期目标**：在16周内掌握C语言核心基础，了解特定方向的技术与框架，具备独立解决中等复杂度问题的能力
- **长期目标**：培养团队协作习惯，激发创新思维，为下学期进阶课程打下基础
- **核心原则**：
	- **能力导向**：以动手实践为主，理论学习为辅
	- **过程监督**：通过老带新机制，实时跟踪新干学习状态
	- **难度梯度**：公共基础项目聚焦基础巩固，自命题项目强调能力跃升

### 分组策略

- **分组逻辑**：
	- 参与人员：21位新干 + 至少6位现干 → 6~10个小组
	- 自由分组，每组3~5人：至少1名现干 + 至少2名新干
- **分组流程**：
	1. 各位参与成员相互沟通协商，进行组队
	2. 组好的队伍填写对应的在线文档，进行登记
	3. 角色分配：
		 - 现干：担任小组导师，技术指导与进度监督
		 - 新干：轮流担任"每日负责人"，培养主动性
		 - 组长：由现干或组内推选
- **注意事项**：
	- 适当进行能力分层，做好分工，确保新干均参与核心任务，不能只写文档或打杂

> 请在团队飞书前往在线文档【项目小组】进行登记。

### 时间线与培训内容

| 阶段             | 时间      | 核心内容                                                              | 交付物                                        |
| ---------------- | --------- | --------------------------------------------------------------------- | --------------------------------------------- |
| **启动准备**     | 第1周     | 团队破冰、C语言基础速训、工具链配置                                   | 个人学习计划表、小组通讯录（填写在线文档）    |
| **公共基础项目** | 期末      | 实现"简易图书管理系统"，主要包括链表操作、树形结构、文件I/O、快速排序 | 11月底：设计文档+核心代码；期末：完整项目提交 | 
| **自命题项目**   | 期末~寒假 | 小组自拟主题（难度>公共项目），经审核后开发                           | 选题通过通知+项目提交                         |
| **过渡与反馈**   | 每周末    | 新干学习日志、现干指导报告                                            | 月度反馈表                                    |

### 老带新机制

- **现干职责**：
	- 每周≥1.5小时1对1辅导（禁止直接写代码）
	- 使用共享文档跟踪任务进度
	- 每周表扬1名进步最大新干
- **新干要求**：
	- 必须参与所有开发环节（设计/编码/测试）
	- 每月提交"技能自评表"，对比初期能力变化
- **双向反馈**：新干匿名评分现干指导质量，确保机制有效落地

## 项目要求

### 公共基础项目

- **项目名称**：简易图书馆管理系统
- **开发语言**：C语言（C23标准，仅限标准库）
- **工具链**：GCC+CMake
- **核心功能**：
	- 图书录入：链表存储，ISBN冲突检测，存储为 JSON 文件
	- 图书查询：ISBN精确搜索，书名/作者模糊搜索
	- 借阅记录：二进制文件持久化（不能仅内存模拟）
	- 排序统计：库存量、借阅量快速排序
	- 数据导出：图书和借阅数据导出为CSV和JSON文件
	- CLI界面：命令行界面交互或TUI，无需GUI
- **技术规范**：
	- 内存管理：无泄漏（Valgrind检测）
	- 模块分离：
		- 文件接口：`store.h`/`store.c`
		- 数据容器：`data.h`/`data.c`
		- 业务逻辑：`logic.h`/`logic.c`
		- 程序入口：`main.c`
	- 代码注释：≥20%代码行
- **交付物**：
	- 设计文档
	- 测试报告
	- 演示视频
	- 项目源代码Git仓库
- **交付时间**：按时间线或最新安排交付

### 自命题项目

- **选题流程**：
	1. 小组提交《选题提案》（含技术方案）
	2. 团队审核组评估难度（A/B/C）
	3. 通过后方可开发
- **难度要求**：
	- 必须实现≥2个进阶功能（如：网络通信+多线程）
	- 禁止选题：操作系统开发、AI模型训练等
- **支持资源**：
	- 选题库
- **交付物**：
	- 设计文档
	- 测试报告
	- 演示PPT
	- 演示视频
	- 项目源代码Git仓库
	- Docker镜像
	- 项目总结
- **交付时间**：按时间线或最新安排交付

> 具体要求和内容细节请关注最新相关通知或咨询领队现干。

## 考核标准

### 项目成果考核（50%）

| 项目类型              | 评分维度     | 评分标准                                     |
| --------------------- | ------------ | -------------------------------------------- |
| **公共基础项目**(30%) | 功能完整性   | • 100%实现核心功能：25分<br>• 每缺1功能扣8分 |
|                       | 代码质量     | • 规范性(15分)+效率(10分)+无Bug(10分)        |
|                       | 文档与演示   | • 设计文档(10分)+演示视频(15分)              |
| **自命题项目**(20%)   | 难度与完成度 | • 达到审核难度：20分<br>• 每少1功能扣10分    | 
|                       | 技术深度     | • 进阶知识应用(15分)+重构合理性(10分)        |
|                       | 团队协作     | • 任务分配(10分)+交流贡献(10分)              |

### 新干参与过程考核（50%）

| 考核维度     | 权重 | 评分标准                                                  | 
| ------------ | ---- | --------------------------------------------------------- |
| **个人贡献** | 30%  | • 代码提交量(Git记录)：15%<br>• 任务完成度(周报记录)：15% |
| **学习成长** | 15%  | • 技能进步(对比测试)：10%<br>• 问题解决(主动提问)：5%     |
| **团队协作** | 5%   | • 沟通主动性(现干评分)<br>• 帮助队友(有记录+2分)          |

> **考核流程**：项目成果（50%现干+30%导师+20%互评）+ 过程数据（100%基于记录）
> **淘汰机制**：缺交技能自评表且未补交，或参与过程得分<40%，取消考核资格


## 注意事项

### 风险防控

- **项目延期**：启动"救火小组"（2名资深现干）提供思路指导
- **能力断层**：为薄弱新干设置"基础强化营"（每周1小时）
- **指导失效**：现干评分低于70分将暂停指导资格

### 资源保障

- **环境配置**：详见【附件1 环境配置清单】
- **知识库**：Q&A知识库（现干解答后存档）
- **资源包**：C语言速查表、Git教程、项目模板

### 长期价值

- 优秀项目开源至团队代码库，作为下届新干学习案例
- 考核结果纳入"技术人才库"，优先推荐校级竞赛
- 获评"最佳新干"者可提前参与真实项目开发
